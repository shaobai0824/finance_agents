## 第 2 部分：商業目標 (Business Objectives) - 「為何做？」

*在投入資源前，我們必須清晰地定義問題、目標使用者和商業價值，避免開發出無人需要的產品。*

| 區塊 | 內容 |
| :--- | :--- |
| **1. 背景與痛點** | *個人投資者面臨資訊過載和專業知識不足的問題* <br><br> **現況痛點**：<br>- 財經資訊散佈在多個平台，難以整合分析<br>- 缺乏專業的理財規劃知識和工具<br>- 投資決策往往基於情緒而非理性分析<br>- 法規合規檢查複雜且容易疏忽<br><br>**影響範圍**：針對有投資需求但缺乏專業知識的個人投資者，估計台灣市場約有 200 萬潛在用戶 |
| **2. 策略契合度** | *此專案支持 AI 賦能金融服務的戰略目標* <br><br> **戰略契合**：<br>- 符合 FinTech 數位轉型趨勢<br>- 展示多 Agent AI 系統在垂直領域的應用價值<br>- 建立 AI 理財諮詢的技術標準和最佳實踐<br>- 為未來擴展至企業級服務奠定基礎 |
| **3. 成功指標 (Success Metrics)** | *量化的關鍵績效指標定義成功標準* <br> - **主要指標**: 系統回應準確率達 85%，用戶滿意度 > 4.2/5.0 <br> - **次要指標**: API 平均回應時間 < 2 秒，月活躍用戶達 1,000 人<br> - **技術指標**: 系統可用性 > 99.5%，錯誤率 < 1% |

## 第 4 部分：範圍與限制 (Scope & Constraints)

*明確定義邊界是專案成功的關鍵。這為 LLM 生成程式碼提供了最重要的「護欄」。*

| 區塊 | 內容 |
| :--- | :--- |
| **功能性需求 (In Scope)** | *核心功能模組清單*<br> - **財務分析模組**: 股票、基金技術面與基本面分析<br> - **理財規劃模組**: 個人風險評估、資產配置建議<br> - **法規諮詢模組**: 投資法規查詢、合規檢查<br> - **新聞整合模組**: 即時財經新聞爬取與分析<br> - **個人資料庫**: 用戶投資組合管理<br> - **RAG 檢索系統**: 智能知識檢索與問答 |
| **非功能性需求 (NFRs)** | *系統必須滿足的品質屬性*<br> - **性能**: API 回應時間需在 2 秒內，系統可支援 100 併發用戶<br> - **安全性**: 敏感資料使用 AES-256 加密，API 使用 JWT 認證<br> - **可用性**: 系統可用性需達 99.5%，支援 7x24 小時運行<br> - **擴展性**: 支援水平擴展，Agent 可獨立部署和升級<br> - **合規性**: 符合個資法要求，投資建議需包含風險警語 |
| **不做什麼 (Out of Scope)** | *明確列出本次不做的事項*<br> - 不提供實際的交易執行功能<br> - 不支援加密貨幣相關分析<br> - 不提供保險和貸款產品建議<br> - 不支援多語言國際化功能<br> - 不包含移動端 APP 開發 |
| **假設與依賴** | *專案基礎假設和外部依賴*<br> - **假設**: 用戶具備基本的網路連線和瀏覽器環境<br> - **假設**: 財經資料來源 API 保持穩定可用<br> - **依賴**: OpenAI/Anthropic API 服務穩定性<br> - **依賴**: 鉅亨網等財經媒體網站的爬取權限<br> - **依賴**: ChromaDB 向量資料庫的效能表現 |
